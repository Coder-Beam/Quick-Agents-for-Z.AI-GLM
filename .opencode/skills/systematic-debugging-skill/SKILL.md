# Systematic Debugging Skill

> 本Skill已本地化到QuickAgents插件中，支持0 Token调试操作。

## 概述

随机修复浪费时间并创建新bug。快速补丁掩盖根本问题。

**核心原则**: 永远在尝试修复之前先找到根本原因。症状修复就是失败。

**违反此流程 = 违反调试精神**

## 铁律

```
没有根因调查就不能提出修复方案
```

如果还没有完成阶段1，就不能提出修复。

## 何时使用

用于任何技术问题:
- 测试失败
- 生产环境bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在这些情况下使用**:
- 时间紧迫时（紧急情况让猜测变得诱人）
- "只需一个快速修复"看起来很明显
- 已经尝试了多次修复
- 之前的修复不起作用
- 不完全理解问题

**不要跳过**:
- 问题看起来简单（简单的bug也有根本原因）
- 匥于赶时间（匆忙保证返工）
- 经理要求立即修复（系统化比盲目尝试更快）

## 四个阶段

必须完成每个阶段才能进入下一个阶段。

### 阶段1: 根因调查

**在尝试任何修复之前**:

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地复现**
   - 能可靠地触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变更可能导致这个问题？
   - git diff、最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名 → API → 服务 → 数据库）:**

   **在提出修复之前，添加诊断工具:**
   ```
   对于每个组件边界:
     - 记录什么数据进入组件
     - 记录什么数据离开组件
     - 验证环境/配置传播
     - 检查每层的状态

   运行一次以收集证据，显示哪里中断
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时:**

   快速版本:
   - 错误值从哪里产生？
   - 什么用这个错误值调用了？
   - 继续向上追踪直到找到源头
   - 在源头修复，而不是在症状处

### 阶段2: 模式分析

**在修复之前找到模式**:

1. **找到工作示例**
   - 在同一代码库中定位类似的可用代码
   - 工作版本和失败版本有什么区别？
   - 复制工作模式

2. **隔离问题**
   - 创建最小复现案例
   - 移除不必要的复杂性
   - 专注于确切的失败点

3. **检查边界条件**
   - 空值？Null？Undefined?
   - 空字符串？空数组？
   - 极端值？负数？
   - 特殊字符？

4. **验证假设**
   - 写下你认为发生的情况
   - 设计测试来验证
   - 如果假设错误 → 修正假设

### 阶段3: 修复实施

**只有在阶段1和2完成后**:

1. **最小化变更**
   - 只改变必要的内容
   - 不要"顺便"重构
   - 保持变更可追溯

2. **写测试先**
   - 首先编写失败的测试
   - 然后编写修复代码
   - 验证测试通过

3. **实现修复**
   - 应用阶段2中识别的模式
   - 保持代码简单
   - 添加解释注释

### 阶段4: 验证

1. **运行完整测试套件**
   - 不只是新测试
   - 确保没有破坏其他功能

2. **手动验证**
   - 在真实场景中测试
   - 检查边缘情况
   - 验证用户体验

3. **提交**
   - 清晰的提交信息
   - 引用相关issue
   - 说明根本原因和修复

## 声明的根本原因示例

```
❌ 错误的: "修复了登录bug"
   - 修复了什么？症状还是根因？

✅ 正确的: "根本原因: 密码验证逻辑未处理空字符串
   - 修复: 在验证前添加空值检查"
```
