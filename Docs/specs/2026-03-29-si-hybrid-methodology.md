## 一、模型概述

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│        阶段-迭代混合模型 (Stage-Iteration Hybrid)           │
│                                                             │
│        瀑布特征 + 敏捷特征 + TDD + 二次确认 + Simplify      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**设计目标**:
- 整合瀑布模型的阶段性清晰

- 强制TDD质量保证
- 利用大模型二次确认特性
- 消除过度设计和幻觉代码

## 二、核心定义

### 2.1 最小功能单元 (Functional Unit)

| 属性 | 标准 |
|------|------|
| 定义 | 一个功能 = 一个单元 = 一次TDD循环 |
| 开发时间 | ≤ 1.5小时 |
| 代码行数 | ≤ 150行 (不含测试) |
| 测试用例 | ≥ 1个 (覆盖率≥80%) |
| 验收标准 | ≥ 1个可量化指标 |
| 提交粒度 | 1个单元 = 1次提交 |

**层级关系**:
```
项目 → 模块 → 功能单元
(1)    (N)    (M)
```

**必须满足**:
- ✅ 独立测试 (单元测试)
- ✅ 独立验收 (可量化标准)
- ✅ 人类可复现 (代码可读性)
- ✅ 原子提交 (一次单元一次提交)

### 2.2 简洁命名规范

**原则**: 能看懂 + 能定位 ≠ 完整表达

| 类型 | 长度限制 | 命名风格 |
|------|----------|----------|
| 变量名 | ≤ 15字符 | camelCase |
| 函数名 | ≤ 20字符 | camelCase |
| 类名/接口名 | ≤ 25字符 | PascalCase |
| 文件名 | ≤ 20字符 | kebab-case |
| 常量名 | ≤ 20字符 | UPPER_SNAKE |

**通用缩写 (无需解释)**:
```
cfg → config
ctx → context
fn → function
msg → message
err → error
val → value
idx → index
len → length
str → string
num → number
```

**命名示例**:
```
❌ getUserAccountBalanceFromDatabase()
✅ getBalance() // 获取用户账户余额

❌ calculateTotalOrderAmountWithDiscount()
✅ calcOrderTotal() // 计算订单总金额(含折扣)
```

## 三、二次确认机制 (Double-Check)

**原理**: 同一Prompt接收两次 → 目标更明确

### 3.1 触发点

| 触发场景 | 说明 |
|----------|------|
| 阶段入口确认 | 进入新阶段时重放目标 |
| 单元开发前确认 | 开始单元开发时重放验收标准 |
| 关键决策确认 | 技术选型/架构变更时 |
| 跨会话衔接确认 | 新会话开始时重载上下文 |

### 3.2 阶段入口确认模板

```markdown
📋 阶段目标确认

当前阶段: [阶段名称]
阶段目标: [目标描述]
输入产物: [上一阶段输出]
预期输出: [本阶段输出]
验收标准: [可量化标准]

请确认以上信息是否正确? (y/n)
```

### 3.3 单元开发确认模板

```markdown
📋 单元目标确认

单元ID: [U-XXX]
单元名称: [功能名称]
所属模块: [模块名称]
功能描述: [具体功能]
验收标准: [可量化指标]
预计时间: ≤1.5小时

请确认以上信息是否正确? (y/n)
```

### 3.4 单元完成确认模板

```markdown
✅ 单元完成确认

单元ID: [U-XXX]
完成状态:
├── [x] 测试用例: N个, 覆盖率: X%
├── [x] 代码行数: N行 (≤150)
├── [x] 注释覆盖: X% (≥15%)
├── [x] Simplify检查: 通过
├── [x] 回归测试: 通过
└── [x] Git提交: [commit-hash]

请确认单元已完成? (y/n)
```

## 四、Simplify优化步骤

**位置**: Green之后, Refactor之前

```
TDD循环: Red → Green → Simplify → Refactor
                        ↑
                     新增步骤
```

### 4.1 Simplify检查清单

| 检查类别 | 检查项 | 动作 |
|----------|--------|------|
| **过度设计** | 未使用的参数 | 删除 |
| | 未使用的返回值 | 简化 |
| | 不必要的抽象层 | 扁平化 |
| | 过度泛化 | 具体化 |
| **幻觉功能** | 需求外的功能 | 标记确认 |
| | 未要求的扩展 | 标记确认 |
| | 假设的边界情况 | 标记确认 |
| **冗余代码** | 重复逻辑 | 提取复用 |
| | 不必要的中间变量 | 内联 |
| | 可删除的条件分支 | 删除 |
| | 过度解构 | 简化 |
| **复杂实现** | 能简单完成却复杂实现 | 简化 |
| | 能用原生API却用库 | 替换 |
| | 能用同步却用异步 | 评估必要性 |
| **代码行数** | 函数>50行 | 拆分 |
| | 文件>200行 | 拆分 |
| | 嵌套>3层 | 扁平化 |

### 4.2 Simplify报告模板

```markdown
📋 Simplify报告 - [单元名称]

原始代码统计:
├── 代码行数: N行
├── 函数数量: N个
├── 最长函数: N行
└── 最大嵌套: N层

Simplify检查结果:
├── 过度设计: [无 / N处]
├── 幻觉功能: [无 / N处]
├── 冗余代码: [无 / N处]
└── 复杂实现: [无 / N处]

简化修改:
├── [修改1描述]
└── [修改2描述]

简化后代码统计:
├── 代码行数: N行 (减少X%)
├── 函数数量: N个
├── 最长函数: N行
└── 最大嵌套: N层

测试验证: ✅ 全部通过
```

## 五、四阶段完整流程

### 阶段1: 需求阶段 (瀑布特征)

| 属性 | 内容 |
|------|------|
| 输入 | 用户需求 / 项目需求.md |
| 流程 | 7层询问模型 → REQUIREMENT.md → 二次确认 → 用户确认 |
| 输出 | REQUIREMENT.md |
| 退出标准 | 用户书面确认需求 |

### 阶段2: 设计阶段 (瀑布特征)

| 属性 | 内容 |
|------|------|
| 输入 | REQUIREMENT.md |
| 流程 | 架构设计 → 模块划分 → UNITS.md → 二次确认 → 用户确认 |
| 输出 | DESIGN.md, MODULES.md, UNITS.md |
| 退出标准 | 用户书面确认设计 |

### 阶段3: 实现阶段 (敏捷 + TDD + Simplify)

| 属性 | 内容 |
|------|------|
| 输入 | UNITS.md |
| 流程 | 每个单元执行8步开发循环 |
| 退出标准 | 所有单元完成 + 全部测试通过 |

### 阶段4: 交付阶段 (敏捷特征)

| 属性 | 内容 |
|------|------|
| 输入 | 完成的所有模块 |
| 流程 | 集成测试 → E2E测试 → 文档 → 性能测试 → 发布 |
| 退出标准 | 用户验收通过 |

## 六、单元开发循环 (≤1.5小时)

| Step | 名称 | 时间 | 内容 |
|------|------|------|------|
| 🔄 | 二次确认 | 2分钟 | 重放单元目标+验收标准 |
| 1 | 单元设计文档 | 5分钟 | 功能描述+接口定义+验收标准 |
| 2 | Red - 测试先行 | 15分钟 | 编写测试用例，确认失败 |
| 3 | Green - 最小实现 | 25分钟 | 编写最少代码，确认通过 |
| 4 | Simplify - 简化优化 | 15分钟 | 运行检查清单，应用简化 |
| 5 | Refactor - 重构 | 10分钟 | 代码结构优化 |
| 6 | 人类可复现检查 | 10分钟 | 命名+可读性+注释检查 |
| 7 | 原子性提交 | 5分钟 | 提交代码+测试+文档 |
| 8 | 回归测试 | 5分钟 | 运行所有相关测试 |
| 🔄 | 二次确认 | 3分钟 | 单元完成确认 |

**总计**: 约95分钟 (≤1.5小时)

## 八、回归测试规范

| 属性 | 内容 |
|------|------|
| 触发时机 | 每个单元完成后 / 每个模块完成后 / 每次代码修改后 |
| 测试范围 | 单元级 → 模块级 → 项目级 |
| 强制要求 | 所有测试必须100%通过 / 不允许跳过失败测试 |
| 失败处理 | 回归失败 = 单元未完成 |
